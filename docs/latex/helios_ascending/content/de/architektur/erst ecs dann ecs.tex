\subsection{Erst ECS, dann ECS - Evolution des Entity-Begriffs}

Die Entwicklung von helios begann mit einem klassischen, objektorientierten Ansatz.\footnote{
    Im Folgenden auch einfach \textit{OOD} (Object-Oriented Design). Das Kürzel \textit{DOD} verwenden wir entsprechend im Kontext des \textit{Data-Oriented Design}.
}
Erst mit zunehmender Komplexität des Projektes und mit den steigenden Anforderungen an eine flexiblere Datenstruktur wurde das \textit{Compound Object} mit dem \textit{Array-of-Structs}-Layout, dass ein GameObject in Meilenstein 4 darstellt, zugunsten eines datenorientierten Ansatz verworfen, der auf dem \textit{Structure-of-Arrays}-Layout\footnote{\textit{SoA}~\cite[162]{Fab18}} basiert (ein Architekturvergleich ist in  Abbildung~\ref{fig:ecs_evolution} dargestellt).

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.8\textwidth]{content/de/architektur/ecs_evolution.png}
    \caption{Evolution des Entity-Begriffs in der Entwicklung von helios.}
    \label{fig:ecs_evolution}
\end{figure}

Die Grundlage dieses SoA-basierten Ansatzes wird gemeinhin als \textit{Entity Component System} (\textit{ECS}) bezeichnet.\footnote{\textit{Fabian} weist darauf hin, dass bereits ein Compound Object, das dem composition over inheritance Prinzip folgt und die Eigenschaften in einer AoS-Struktur soeichert, als ECS bezeichnet werden kann~\cite[84]{Fab18}. Wir verwenden den Begriff immer im Kontext eines SoA-Layouts, womit gleichzeitig der Schwerpunkt nicht allein auf der Konzeption von Objekten und Eigenschaften liegt, sondern auch auf einer möglichst effizienten Anordnung der Daten im Speicher.}

Die Idee hinter ECS liegt in der Trennung von Daten und Verhalten und hat in der Spieleentwicklung prominente Vertreter, darunter das bereits erwähnte Unity Dots sowie die Open Source Projekte Bevy (Rust)~\cite{BevyProject}, EnTT~\cite{EnttProject} und Flecs~\cite{FlecsProject} (beides C++).
ECS wird aber auch in anderen Domänen in performancekritischen Bereichen eingesetzt, wie  bei GPU beschleunigter, nebenläufiger  Batch-Simulation zum Training von KI-Agenten ~\cite{SRXB+23}

Es ermöglichst ein typenloses Datenmanagement und spart durch den Verzicht auf (tiefe) Vererbungshierarchien VTable-lookups zur Laufzeit ein.


\subsubsection{SparseSet als Datenhaltung in helios.ecs}

Der Zugriff auf den CPU-nahen Level-n Cache ist um ``etwa 10 bis 100 mal schneller als die Zugriffszeiten gängiger Hauptspeicher``~\cite[S. 5]{Man20}~\cite[79]{HP17}.
In den Level-n Cache werden nach bestimmten Heuristiken Daten bewegt, auf die die CPU oft zugreifen muss (Instruktionen, Programmdaten).
Dies erfolgt in Form zusammenhängender Speicheradressen - sogenannte \textit{cache lines}: Typischerweise umfassen diese 32-64 Byte große Blöcke~\cite[531]{TB23}.
Gelingt es, zusammenhängende Daten auch in zusammenhängende Speicheradressen zu schreiben, und diese dicht aneinanderliegen, kann ein Programm davon profitieren, dass diese zusammenhängenden Speicherbereiche auch als CacheLine zur Verfügung stehen, was den Datenzugriff beschleunigt: Anstatt teure Zugriffe auf den Hauptspeicher durchführen zu müssen, können in performancekritischen Anwendungen (wie Echtzeitsystem) durch ein vom Entwickler intentionale Speicherlayout Zugriffe auf die Daten um ein Vielfaches beschleunigt werden.\\

\noinndet
Wird hingegen ein angefordertes Speicherwort nicht im Cache gefunden, wird ein \textit{cache miss} ausgelöst: In Folge dessen ist ein Zugriff auf den Hauptspeicher nötig.

Erste Iterationen von Helios begünstigten diese Cache Misses, da die GameWorld die enthaltenen GameObjects als \texttt{std::vector<unique\_ptr<GameObject>>} verwaltet hat, und diese verwalteten wiederum - in einem Array-of-Structs-Ansatz als \textit{Compound Objects} - die assoziierten Komponenten als \texttt{std::vector<unique\_ptr<Component>>}.
Iteriert ein System über alle GameObjects, um eine bestimmten Komponenten-Instanz vom Typ $\mathcal{A}$ zu aktualisieren, muss das System über alle Objekte vom Typ GameObject laufen, und gleichzeitig einen teuren RTTI-cast\footnote{\textit{Runtime-Type-Information}} auf den Komponententypen $\mathcal{A}$ ausführen.
Ein illustratives Codebeispiel ist in Listing~\ref{lst:gameobject_cache_miss} dargestellt.



\begin{lstlisting}[style=c++style,
    caption={Codebeispiel für die Iteration über alle GameObjects, um Komponenten eines bestimmten Typs $\mathcal{A}$ zu finden (Meilenstein 3). Die Implementierung begünstigt Cache Misses, da die angefragten Daten nicht in zusammenhängenden Speicherbereichen liegen.
Für den performancekritischen Hot Path in GameEngines entstehen zudem durch das \textit{Pointer Chasing} \texttt{world} $\rightarrow$ \texttt{gameobject} $\rightarrow$ \texttt{component}~\cite{HKSN+16} und den RTTI-cast unverhältnismäßig teure Operationen.},
    label=lst:gameobject_cache_miss]
    for (auto& go: gameWorld.gameObjects()) {
         if (go->guid() != guid) {
             continue;
         }
         for (const auto& component : go->components()) {
                if (auto* comp = dynamic_cast<A*>(component.get())) {
                    return comp;
                }
            }
    }
\end{lstlisting}

Eine alternative Implementierung auf Basis von dem in helios implementierten ECS-System ist in Listing~\ref{fig:ecs_cache_friendly} dargestellt.
Der \texttt{EntityManager} verwaltet einen Vektor mit Zeigern auf \texttt{SparseSets} die gemäß dem Structure-of-Arrays Ansatz jeweils Komponenten eines Typs vorhalten.
Die Indexierung findet auf Grundlage eines eigens implementierten TypeIndex statt, der Indizes beginnend bei $0$ für die CompileTime-Typen liefert.
Da Komponenten ohnehin ausschließlich in Sparse Sets über den EntityManager gespeichert werden, ist somit eine dichte Anordnung der Komponenten-Arrays garantiert, und der Zugriff auf einen Sparse Set des Typs $\mathcal{A}$ erfolgt konstanter Zeit $O(1)$:

\begin{verbatim}
    auto* sparseSet<A> = EntityManager::getSparseSet<A>()
\end{verbatim}

\noindent
Das übergebene Template-Argument $A$ wird intern über einen entsprechenden \texttt{TypeIndex} in ein \texttt{size\_t}-Wert umgewandelt.
Insgesamt kann damit die o.a. Implementierung wie in Listing~\ref{fig:gameobject_component_access} vereinfacht werden.


\begin{lstlisting}[style=c++style,
    caption={Zugriff auf eine Komponente eines GameObjects mit der ECS-Implementierung in helios (Meilenstein 5).},
    label=lst:gameobject_component_access]
    auto gameObject = gameWorld.find(entityHandle);
    if (!gameObject) {
        return nullptr;
    }
    return gameObject->get<A>();
\end{lstlisting}

Unter der Annahme, dass fortlaufende Speicherbereiche im Cache vorgehalten werden, kann auch der Zugriff während einer Iteration über Komponenten beschleunigt werden.
Dies ist gerade bei GameEngines in der GameLoop von Bedeutung, wenn tausende Entities in einem Frame aktualisiert werden müssen.
Typisch ist dies unter anderem für Physik-Systeme, die die Translation von den beteiligten Objekten anhand einer \texttt{TransformComponent}-Komponente aktualisieren, wie in Listing~\ref{lst:gameobject_transform_component} beispielhaft angegeben.

\begin{lstlisting}[style=c++style,
    caption={Codebeispiel ecs Sparse set},
    label=lst:gameobject_transform_component]
    auto* components = gameWorld.entityManager().getSparseSet<TransformComponent>();
    for (auto& cmp : components) {
        cmp->translate(vec3Position);
    }
\end{lstlisting}

Insgesamt ist ein ECS also nicht nur ein Paradigma für die Trennung von Verhalten und Daten und eine Anforderung an den Entwickler, fachlich saubere Entwürfe von Komponenten zu erstellen: Ihre Stärke spielen ECS-Implementierungen vor allem dann aus, wenn die Daten effizient im Speicher angeordnet werden, wie Cox et al. in~\cite{CWVW+25} zeigen.\footnote{
Weitere Verfeinerungen von Speicherlayouts können über Archetypes / Chunks realisiert werden, die eine feste Anordnung von Komponenten vorsehen, um den Zugriff auf häufig benötigte Kombinationen von Komponenten zu beschleunigen, wie in Unity Dots~\cite{UnityChunks}.
Weitere Optimierungen sind über die Anordnung der Daten innerhalb der Structs / Klassen zu erreichen, wie Gregory in ~\cite[159-161]{Gre19} darstellt.
}
