\subsubsection{Manager}

Manager in helios diesen zur Orchestrierung von verschiedenen Aktionen innerhalb der GamLoop. Sie werden zum größten Teil auch als CommandHandler eingesetzt und sind dafür verantwortlich in einem Frame Commands zu sammen und zu einem bestimmten Zeitpunkt, der durch die Gameworld über flushManagers() ausgelöst wird, ihre Logik auszuführen bzw. die gesammteln Commands - wenn möglich gebatcht - zur Ausfürhung zu bringen.

Manager in helios besitzen keinen Basistypen.
Auch hier werden wieder \texttt{Concepts} (\texttt{IsManagerLike}) und \texttt{EngineRoleTags} (\texttt{ManagerRole}) zur Kennzeichungn von Klassen verwendet, die als Manager fungieren sollen.

helios implementiert zur Zeit u.a. folgende manager:

\begin{itemize}
    \item \texttt{HealthManager}: Verarbeitet Befehle zur Anwednung von Scahaden auf die in der GameWorld aktiven GameObjects, bspw ausgelöst durch Kollisionen.
     Aktualisierungen der \texttt{HealthComponent} der betreffenden Objekte werden dabei über Events an die nächste Phase kommuniziert, was eine VErarbeiotung als grafische Effekte oder Audio-Elemente erlaubt. Der Manager sorgt auch dafür, dass dass die Objekte eine \texttt{DeadTagComponent} erhalten, um bestimmte Systeme (wie Bewegungs) nicht mehr in einem Frame zu berücksichtigen.
    \item \texttt{ScorePoolManager}: Verwaltet die \texttt{ScorePools} in helios, die ddafür verwendet werden, um Punkte bzw. Belohnungen zu sammeln. Die ScorePools sind abstrakte Punktesysteme, die klassische Punktestände repräsentieren können und über ScorePoolIds an GameObjects gebunden werden könenn, so dass verschieden GameObjects über eigene, verschiedee Punktestände verfügen, was zum Beispiel die Berechnung von Erfahrungspunkte über einen solchen Pool erlaubt. Das Scoring selber ist als Mechanik in \texttt{helios.engine.mechanics.scoring} implementiert. Der Manager integriert diese auf breiter Ebene in der GameWorld.
    \item \texttt{GameObjectPoolManager}: Verwaltete die verschiedenen Obejtkpool der GameObjects, um eine Erstellung von Spielobjeken zur Laufzeit mögkchst zu vermeiden. Der manager verwaltet hierbei eine LIste von PoolConfigurationen, die eine festgelegte Zahl von GameObjects basierend auf einem Prefab enthalten, und bietet Fassadenfunktionen zur Rückgabe und Bereitstellung der in dem Pool enthaletnen Objecte an.
    \item \texttt{Spawnmanager}: Entgegen vieler anderer Manager, die auch als CommandHandler fungieren, arebitet der  GameObjectPoolManager nicht als Befehlsverarbeiter zur Verwaltung der Objekte der Pools - diese Aufgabe übernimmt der SpawnManager, der von dem SpawnSystem auf Basis verschiedener Regeln und Bedingungen GameObjects in die Spielwelt spawnt (\texttt{SpawnCommand}) - oder sie aufräumt und zurück in die GamePools legt (\texttt{DespawnCommand}).
    Hierzu besitzt der Spawnmanager einen Zeiger auf den PoolManager, um die entsprechenden Funktionen zur Bereitstellung bzw. Rückgabe der GameObjects aufzurufen.
    \item \texttt{StateManager}: Eine besondere Veranwtortung  besitzt der StateManager, der die Verwaltung der verschienden States übernimmt. Hierzu ist der StateManager teamplate-basiert implementiert und ist damit nicht an vordefinierte States gebunden. Auf das StateManagement gehen wir kurz als gesondertes Konzeot in Kapitel~\ref{sec:state_management} ein.
\end{itemize}

Daneben finden sich noch \texttt{WorldLifecycleManager} zur Verwaltung von Lifecycle-Commands wie einem \texttt{reset()} der GameWorld, \texttt{TimerManager} zur Verwaltung von Spielmechaniken, die eine Zeitmessung benötigen, \texttt{UiActionCommandManager} zur Verarbeitung von Ui-Commands (wie aktivierung von Menüoptionen).
