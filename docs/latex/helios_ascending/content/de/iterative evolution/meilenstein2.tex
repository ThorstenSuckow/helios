\section{Meilenstein 2: Controller-Input und die Evolution des GameObjects}


Meilenstein 2 markierte die erste funktionale Erweiterung der Architektur, die ein Rendering in Abhängigkeit von Benutzereingaben (Gamepad-Input) ermöglichte.
Hierfür wurden die Klassen \texttt{GameWorld} und \texttt{GameObject} eingeführt.
Deren Semantik blieb zwar bis zur finalen Version erhalten, sie durchliefen jedoch im Prozess eine große strukturelle Transformation, von Datencontainern, die Geschäftslogik kapseln, hin zu leichtgewichtigen Fassaden-Objekten für den Zugriff auf darunterliegende, Template-basierte Registries und einem datenorientierten \textit{Entity Component System} (im Folgenden ``ECS``).\\


Der im vorherigen Abschnitt angesprochene architektonische Umbruch des Rendering Systems betraf die Repräsentanten der Viewports~\cite[40]{SWH15}, welche als statische, eins-zu-eins zum Fenster korrespondierende Render-Ziele verstanden wurde.
Diese wurden so angepasst, dass innerhalb der Engine die Unterstützung mehrerer Viewports über einen  \texttt{RenderTarget} (Abstraktion eines Framebuffers~\cite[S. 329]{HDMS+14}) ermöglicht wird.
Dies vereinfacht die Darstellung unterschiedlicher Szenen oder Szenenausschnitte, die von verschiedenen  Kameras beobachtet werden.
Als positiver Seiteneffekt erleichterte diese Abstraktion die Einführung von UI-Elementen in Meilenstein 5.\\

Die in Abbildung~\ref{fig:gamecontrollerinput} dargestellte Demonstration zeigt die Funktionalität der Eingabeverarbeitung, die über die Klasse \texttt{InputManager} realisiert wird.
Der \texttt{InputManager} liegt in der äußersten Anwendungsschicht und wird in der \textit{GameLoop} angefragt, damit Gamepad-Eingaben als Transformationen der dargestellten Objekte umgerechnet werden können.
Weitere Abstraktionsebenen fehlten an dieser Stelle, Szenenknoten wurden mittels \texttt{setScale()} und \texttt{setTranslation()} direkt modifiziert, Farbänderungen durch direkte Aufrufe auf mit den Knoten verbundenen \textit{Renderables} realisiert.\\

In dieser Phase zeigte sich auch, dass der ursprüngliche Entwurf des Rendering-Systems zu komplex war.
Eine Restrukturierung hin zu einer flacheren Hierarchie (siehe Abbildung~\ref{fig:renderable}) ermöglichte eine effiziente Wiederverwendung von Material-, Mesh- und Shader-Instanzen, die in einem \texttt{RenderPrototype} als \textit{Aggregate Root}~\cite[S. 128]{Eva03} vereint wurden, teilbar von mehreren \texttt{Renderables}.
Optionale Konfigurationen der \texttt{Renderables} in Form von kleinen Konfigurationsstrukturen / DTOs erlauben dabei individuelle Overrides für einzelne Instanzen, die damit konsequenterweise den \textit{extrinsischen Zustand} eines grafischen Objektes~\cite[S.196-197]{GHJV94} speichern.

Mit diesem Meilenstein wurde das Modul \texttt{helios.engine.game} zusammen mit der Klasse \texttt{GameWorld} eingeführt, welche als zentraler Container und zur Verwaltung alle in der Spielwelt befindlichen \texttt{GameObjects} verantwortlich gewesen ist, wobei die Umsetzung zu diesem Zeitpunkt noch stark von einem klassischen OOP-Ansatz geprägt ist, wie das in Listing~\ref{lst:gameobject_v1} dargestellte Interface auszugsweise verdeutlicht.

\vspace{5mm}
\begin{lstlisting}[style=c++style,
    caption={
        Auszug aus dem Interface der \texttt{GameWorld} aus Meilenstein 2.
        Die Aktualisierung der Spielobjekte geschah hier noch über eine zentrale \texttt{update()}-Methode, die innerhalb der \textit{GameLoop} aufgerufen wurde.
        Der Zugriff auf die \texttt{GameObjects} erfolgte hashbasiert über ihre jeweilige GUID.
    },
    label=lst:gameworld_v1]
    /**
     * @brief Updates all GameObjects in the world for
     *        the current frame.
     */
    void update(float deltaTime) noexcept;

    /**
     * @brief Adds a GameObject to the world.
     */
    void addGameObject(std::unique_ptr<GameObject> gameObject);

    /**
     * @brief Finds a GameObject by its unique identifier.
     */
    GameObject* find(const Guid& guid);
\end{lstlisting}
\vspace{5mm}

Obwohl die API auch im aktuellen, abgegebenen Zustand die gleiche Semantik benutzt, weicht die Implementierung sehr stark von der hier beschriebenen Umsetzung ab:
Auf hash-basierte Zugriffe über \texttt{std::unordered\_map<T>} wird in dem Engine-Modul weitestgehend verzichtet, die Steuerung der in einem Frame aktiven Systeme erfolgt über eine Abstraktion des \textit{GameLoop}-Konzepts in zustandsabhängige Phasen und Pässe mit individuellen Commit-Points, um deterministische Nachrichtenverarbeitung zu realisieren (siehe Abschnitt~\ref{sec:meilenstein_4}).\\

Weitere Implementierungen in diesem Meilenstein betrafen unter anderem:

\begin{itemize}
    \itemsep0.5em
    \item Tooling-Overlay mit Hilfe von \textit{Dear ImGui}~\cite{DearImgui} zur einfacheren Auswertung von Log-Nachrichten und der gezielten Konfiguration von \texttt{GameObject}-Parametern zur Laufzeit (siehe Abbildung~\ref{fig:spaceshipcontrol}).
    \item \texttt{CameraSceneNode} als spezialisierte \textt{SceneNode} mit \texttt{lookAt()}-Implementierung~\cite[S. 128-129]{PJH23} sowie Bitmaske zur Spezifizierung der zu vererbenden Transformationsdaten in Eltern-Kind-Szeneknoten-Beziehungen.
\end{itemize}
