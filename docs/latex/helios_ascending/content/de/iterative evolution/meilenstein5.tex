\section{Meilenstein 5: ECS und UI-Architektur}\label{sec:meilenstein_5}

Ziel von Meilenstein 5 war die Umsetzung und Integration von Gameplay-Mechaniken in das zugrundeliegende Framework.
Da \textit{helios} zum Ende des vorangegangenen Meilensteins bereits sehr stabil lief, wurde zunächst die mit Meilenstein 3 eingeleitete gedankliche Umstellung auf eine ECS-Architektur nun auch praktisch vollzogen.

Wir sahen hier zunächst Bedarf, die Datenhaltungsschicht der \textit{GameWorld} umzustrukturieren, was ein Entkernen der direkt verwalteten \texttt{GameObject}s, der \texttt{Manager}-Klassen sowie der \texttt{CommandHandler} und \texttt{CommandBuffer} bedeutete.
Gemessen an dem konzeptionellen Bruch, den die Einführung des ECS-Konzepts in das Projekt mit sich brachte, musste vergleichsweise wenig an der öffentlichen API angepasst werden: Die Änderungen beschränkten sich weitgehend auf die tieferen, gekapselten Schichten; die öffentlichen Schnittstellen von \textit{GameWorld} und \textit{GameObject} blieben weitestgehend unverändert.\footnote{
Die Stabilität der API führen wir an dieser Stelle auf die bereits im Prototypen-Dokument beschriebenen SOLID-Prinzipien und die konsequente Kapselung von Funktionalitäten zurück, ganz im Sinne des \textit{Information Hiding}~\cite{Par72}.
}

Ein hoher Migrationsaufwand entstand jedoch durch den Austausch der bislang in \textit{helios} verwendeten GUIDs zur Identifikation von Spielobjekten, die sich durch alle Schichten der bisher umgesetzten Engine und ihrer Subsysteme zogen.
Da diese GUIDs nun durch einen \texttt{EntityHandle} ersetzt wurden - der einem zusammengesetzten Schlüssel entsprach -, waren die Änderungen zwar zahlreich, aber überwiegend von syntaktischer Natur.

Neben der Umstellung der Datenhaltung erwies sich in diesem Meilenstein die Integration eines User Interface als weitaus folgenreicher.

\subsubsection{UI-Elemente als Scope Creep}
Das für die Steuerung des Spiels notwendige User Interface in Form eines Heads-Up-Display (\textit{HUD}) sollte Menüoptionen zum Starten, Pausieren und Beenden des Spiels erhalten.

Die Fertigstellung hat sich dadurch wesentlich verzögert: Für die Textdarstellung folgen wir einer einfachen Umsetzung nach de~Vries~\cite[S.~449-457]{Vri20}, der die Anzeige von Text als Glyph-Rendering mittels FreeType~\cite{freetype} umsetzt.
Das Text-Rendering wird in der Render-Pipeline in einem auf das Mesh-Rendering folgenden Durchlauf berücksichtigt.
Texte werden dabei als spezialisierte \textit{Renderables} behandelt, die neben der Zuordnung der Font-Ressource auch die Konfiguration von Skalierungsfaktor und Textfarbe erlauben.\\

Eine semantische Erweiterung des bisher primär als (inter-)aktiven verstandenen \textit{GameObject}-Begriffs ergibt sich durch die Behandlung von Texten als eben solche Objekte.
Auch diese sind bei \textit{helios} in der \textit{GameWorld} verankert (siehe~\ref{lst:gameobject_text}), wodurch wir eine weitere Abstraktionsschicht eingespart haben.
Komponenten, welche die für die Anzeige benötigten Daten vorhalten (z.,B.~Anzahl der Spielerleben, ScorePool-IDs, Timer), werden über ein einfaches Binding-System pro GameLoop-Iteration abgefragt und zur Anzeige gebracht.


\begin{lstlisting}[style=c++style,
  caption={Implementierung eines einfachen Score-Textes, der die Punktzahl des Spielers anzeigt. UI-Elemente werden analog zu aktiven, mit der Spielwelt verbundenen Objekten modelliert und entsprechend konfiguriert.},
  label=lst:gameobject_text]
auto scoreText = GameObjectFactory::instance()
    .gameObject(gameWorld)
    .withRendering([&](auto& rnb) {
        rnb.textRenderable()
           .formattedAsNumber("Score: {:.0f}")
           .fontScale(1.0f)
           .color(helios::util::Colors::WhiteSmoke);
    })
    .withObserver([&](auto& ob) {
        ob.observe()
          .runningScore(ScorePoolId{"PlayerOne_ScorePool"});
    })
    .withUiTransform([](auto& tb) {
        tb.transform()
          .pivot(Anchor::TopRight)
          .anchor(Anchor::TopRight)
          .viewport(ViewportId{"hudViewport"})
          .offsets({60.0f, 50.0f, 0.0f, 0.0f});
    });
\end{lstlisting}

Neben der Textanzeige erforderte die Umsetzung der Spielmenüs außerdem eine Evaluierung von Spielzuständen, die in \textit{helios} über endliche Automaten realisiert wurde.
Die Implementierung des Moduls \texttt{helios.engine.state} folgt einem generischen Muster auf Basis von Templates.
Damit ist die Definition weiterer Zustandsautomaten möglich, um beispielsweise hierarchische Automaten für komplexere Spielsteuerungen zu realisieren (siehe Abschnitt~\ref{sec:state_management}).\\

Mit der Unterstützung von endlichen Automaten wurde die \texttt{GameLoop} zusätzlich um die Möglichkeit erweitert, Pässe über eine Bitmaske den Zuständen zuzuordnen, in denen sie ausgeführt werden sollen.

Weitere Features, die in diesem Meilenstein hinzugefügt wurden, sind weniger architektonischer Natur, sondern betreffen überwiegend Gameplay-Mechaniken, unter anderem die Einführung von \textit{Timer}-Systemen für zeitbasierte Level sowie die Umsetzung von \textit{Player Lives} zur Berücksichtigung in entsprechenden Regelsystemen und einem \textit{Game Over}-Zustand.
