# helios v0.5.0-milestone5

**Release Date:** March 30, 2026  
**Project:** helios Game Framework  
**Milestone:** ECS Refactor, Typed Commands, State Management & Gameplay Systems

---

## Overview

This marks the fifth milestone release (**milestone_5**) of the helios project. The release replaces the inheritance-based entity model with a SparseSet-based ECS, introduces a compile-time typed command system, a type-indexed resource registry, template-based state management, and a full set of gameplay mechanics (health, damage, scoring, timers, lives, UI). The `scoring_demo` example demonstrates all systems working together in a playable prototype.

---

## Major Features

### SparseSet-Based ECS Architecture

The entity model has been rewritten from scratch. GUID-based identification is replaced by versioned `EntityHandle` references (index + version). Components are stored in paged `SparseSet<T>` containers with O(1) lookup and dense iteration. `GameObject` is now a lightweight facade (~16 bytes, safe to copy by value).

| Class | Purpose |
|-------|---------|
| `EntityHandle` | Versioned entity reference (8 bytes) |
| `EntityId` / `VersionId` | Strongly-typed identifiers |
| `EntityRegistry` | Handle allocation, validation, and recycling |
| `EntityManager` | Component storage via `SparseSet<T>` per type |
| `SparseSet<T>` | Paged sparse array with swap-and-pop removal |
| `View` | Component-based entity queries with `whereEnabled()` filter |
| `System` | Type-erased wrapper (replaces inheritance-based `Updatable`) |

Components no longer inherit from a `Component` base class. Lifecycle behavior is detected at compile time via traits (`HasClone`, `HasReset`, `HasEnable`, `HasDisable`). The `ComponentReflector` and `ComponentOpsRegistry` provide runtime function pointers for clone, reset, enable, and disable operations.

The `Active` / `Inactive` tag components control entity visibility. Systems use `View::whereEnabled()` to iterate only over active entities.

### Compile-Time Typed Command System

The runtime `unordered_map`-based command dispatch is replaced by a compile-time typed approach:

| Class | Purpose |
|-------|---------|
| `TypedCommandBuffer<...Ts>` | Per-type queues stored in `std::tuple<std::vector<T>...>` |
| `TypedCommandHandler<T>` | Type-safe handler interface |
| `EngineCommandBuffer` | Pre-configured facade with all engine command types |
| `CommandBufferRegistry` | Type-indexed storage of `CommandBuffer` instances |
| `CommandHandlerRegistry` | Type-indexed storage of command handlers |

Commands are plain structs (value types). The `ExecutableCommand` concept allows commands to self-execute via an `execute(UpdateContext&)` method. Flush order is deterministic, following the template parameter order: combat, motion, scoring, spawn, state, UI, timing, lifecycle.

### Resource Registry

A type-indexed service locator for engine resources with O(1) access:

| Class | Purpose |
|-------|---------|
| `ResourceRegistry` | Central storage for Managers, CommandBuffers, handlers |
| `ResourceTypeId` | Compile-time type identifier |
| `ErasedUnique` | Type-erased unique ownership wrapper (16 bytes) |

Registration and access:

```cpp
auto& poolManager = gameWorld.registerManager<GameObjectPoolManager>();
auto& manager     = gameWorld.manager<GameObjectPoolManager>();
```

The `ResourceRegistry` is hidden behind commit points and not directly accessible from `UpdateContext`.

### Template-Based State Management

Generic `StateManager<TState>` template eliminates code duplication:

| Class | Purpose |
|-------|---------|
| `StateManager<TState>` | Template base for state machines |
| `GameStateManager` / `MatchStateManager` | Concrete specializations |
| `GameFlowSystem` / `MatchFlowSystem` | State transition processors |
| `StateTransitionContext<T>` | Type-safe transition data |
| `Session` | Tracks game/match state, player entity, viewport IDs |

`GameState` and `MatchState` are bitmask enums. Transition guards validate state changes. Lambda-based listeners support enter-only and transition-only callbacks. `StateToIdMapPair` maps state combinations to typed identifiers (viewport IDs, menu IDs).

### Builder DSL

`SpawnSystemFactory` provides a fluent builder for spawn configuration:

```cpp
SpawnSystemFactory::instance()
    .pool(poolManager, IdConfig::EnemyPool, IdConfig::EnemyPrefab, 50)
    .profile(IdConfig::EnemySpawnProfile)
        .placer(std::make_unique<RandomSpawnPlacer>(bounds))
        .moveInitializer(DirectionType::Random)
    .scheduledBy()
        .timerWithAvailabilityCondition(2.0f, IdConfig::EnemyPool)
        .fixedAmount(3)
    .commit(spawnManager);
```

Additional builders: `HealthBuilder`, `ScoringBuilder`, `ObserverBuilder`, `UiTransformBuilder`, `CombatConfig`, `TextRenderableConfig`.

### Bootstrap Factory

`makeGameWorld()` returns a pre-configured `GameWorld` + `GameLoop` pair:

```cpp
auto [gameWorldPtr, gameLoopPtr] = makeGameWorld();
```

Pre-registers `EngineCommandBuffer`, `WorldLifecycleManager`, `GameStateManager`, `MatchStateManager`, and tracked states.

### Scene and Viewport

| Class | Purpose |
|-------|---------|
| `SceneToViewportMap` | Associates scenes with viewports |
| `SceneRenderingSystem` | Automated viewport-based scene rendering |
| `StateToViewportPolicyUpdateSystem<Lft, Rgt>` | State-driven viewport activation |
| `ViewportId` / `SceneId` | Strongly-typed identifiers |

### Component Reflection and Bootstrap

Per-module component registries are aggregated in `helios.engine.bootstrap`:

```cpp
registerAllComponents();  // registers all component types
auto [gameWorldPtr, gameLoopPtr] = makeGameWorld();
```

`ComponentReflector` stores function pointers for `clone`, `reset`, `enable`, `disable` per component type. The `ComponentOpsRegistry` provides lookup by type.

| Class | Purpose |
|-------|---------|
| `ConceptModelRegistry` | Type-indexed storage of type-erased wrappers |
| `SystemRegistry` | Uses `ConceptModelRegistry` for system storage |
| `ManagerRegistry` | Uses `ConceptModelRegistry` for manager storage |

## Breaking Changes

This release contains a large number of breaking changes. The most significant ones are grouped by area.

### Entity Model

- `GUID`-based identification replaced by `EntityHandle` (index + version)
- `GameObject` is now a lightweight facade (~16 bytes, copy by value)
- `Component` base class removed; components use compile-time traits
- All component `explicit` copy constructors removed (aggregate-style)
- `Updatable` base class removed; systems use type-erased `System`
- `CloneableComponent` base class removed
- `GameObjectFilter` and `helios.engine.ecs.query` module removed

### Command System

- `CommandBuffer` is now abstract; concrete implementation is `EngineCommandBuffer`
- `TargetedCommand`, `WorldCommand`, and all command base classes removed
- All command dispatchers removed (11 dispatchers replaced by `TypedCommandHandler<T>`)
- `Aim2DCommand` and `ShootCommand` are now self-executing plain structs
- Commands are flushed in deterministic template parameter order

### GameWorld and UpdateContext

- `GameWorld::addManager<T>()` replaced by `registerResource<T>()`
- `GameWorld::getManager<T>()` replaced by `manager<T>()` (returns reference)
- `Manager::flush()` takes `UpdateContext&` instead of `GameWorld&`
- `Manager::onAdd()` removed; use `Manager::init()` instead
- `Manager::gameWorld_` protected member removed
- `UpdateContext` no longer exposes `GameWorld` reference
- `GameLoop` no longer owns `CommandBuffer`; it lives in `ResourceRegistry`
- `GameLoop::update()` signature changed

### GameLoop

- `Phase::addPass()` requires state template parameter: `.addPass<GameState>(GameState::Any)`

### Rendering

- `Renderable` refactored; `RenderableComponent` uses `MeshRenderable*`
- `SnapshotItem` uses raw pointer instead of `weak_ptr`
- `RenderPass` uses non-owning pointers; manual loop replaced by `SceneRenderingSystem`
- `helios.rendering.model` renamed to `helios.rendering.material`
- `makeRenderCommand` emits directly to render queue

### State

- `GameStateManager` / `MatchStateManager` derive from `StateManager<TState>`
- `GameState::Start` split into `Booted`, `MatchReady`, `Title`

### Identifiers

- `Viewport` constructor requires `ViewportId`
- `Scene` constructor requires `SceneId`
- `GameObjectPoolConfig` uses `PrefabId` instead of `EntityHandle`
- `ViewportId` replaces raw viewport references

### Renames

| Before | After |
|--------|-------|
| `DeathEvent` | `HealthDepletedEvent` |
| `LastAttackerComponent` | `LastDamageComponent` |
| `GameTimerObserverComponent` | `GameTimerBindingComponent` |
| `TimerState::Started` | `TimerState::Running` |
| `PushEvent` | `PhaseEvent` |
| `CollisionBehavior::Despawn` | `CollisionBehavior::PassEvent` |
| `helios.engine.core.data` | Dissolved into domain modules |
| `helios.core.buffer` | `helios.core.container.buffer` |
| `helios.engine.Bootstrap` | `helios.engine.bootstrap` |

## Migration Guide

### Entity creation

```cpp
// Before (Milestone 4)
auto& go = gameWorld.createGameObject();
go.add<Move2DComponent>();

// After (Milestone 5)
registerAllComponents();
auto [gameWorldPtr, gameLoopPtr] = makeGameWorld();
auto go = GameObjectFactory::instance()
    .gameObject(*gameWorldPtr)
    .withMotion([](auto& mcb) {
        mcb.move2D().speed(30.0f);
    })
    .make();
```

### Manager registration

```cpp
// Before
gameWorld.addManager<SpawnManager>(std::make_unique<SpawnManager>());
auto* mgr = gameWorld.getManager<SpawnManager>();

// After
auto& mgr = gameWorld.registerManager<SpawnManager>();
auto& mgr = gameWorld.manager<SpawnManager>();
```

### Command handling

```cpp
// Before
gameWorld.registerDispatcher<SpawnCommandDispatcher>();

// After
// Managers register their own TypedCommandHandler<T> during init()
// No manual dispatcher registration needed
```

### System registration

```cpp
// Before
gameLoop.phase(PhaseType::Pre).addPass().addSystem<Move2DSystem>();

// After
gameLoop.phase(PhaseType::Pre)
    .addPass<GameState>(GameState::Running)
    .addSystem<Move2DSystem>();
```

- `zFar` assertion in camera setup
- Move operation incorrectly applied to pointer
- Font cache lookup assertion message

## Known Limitations

- Single-threaded execution model
- Developed and tested on Windows (MSVC 2022) only; Linux and macOS are untested
- Grid collision detection assumes uniform cell sizes
- Text rendering caches ASCII 0--127 only
