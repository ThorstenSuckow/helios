**Release Date:** March 30, 2026  
**Project:** helios Game Framework

## Overview

This marks the sixth and final milestone release (**milestone_6**) of the helios project in the winter semester 25/26. The release accompanies the submission of the *helios engine document*, which provides a comprehensive academic analysis of the architecture and development process behind the helios game engine.

No functional code changes are included in this release. All engine features were finalized in [v0.5.0-milestone5](v0.5.0-milestone5.md).

### Architecture Chapters

#### System Layering and Module Structure
Documents the `helios.engine` module hierarchy (`builder`, `common`, `core`, `ecs`, `mechanics`, `modules`, `runtime`, `tooling`) and naming conventions for deeper layers (`commands`, `components`, `events`, `systems`, `types`).

#### ECS — Entity Component System
- **Sparse Sets**: Dense storage layout (SoA) with O(1) access and O(N) iteration, benchmark comparison against the previous AoS approach
- **EntityHandle**: Generational indexing with `EntityId` + `VersionId`, tombstones, free list management via `EntityRegistry`
- **Components**: No base class, compile-time hooks via Concepts (`onAcquire`, `onRelease`, `onActivate`, `onDeactivate`, `enable`, `disable`, `onClone`)
- **GameObject**: Lightweight 16-byte facade (pointer + handle) replacing the 96-byte Milestone 4 implementation
- **Systems**: Tag-based role identification via `EngineRoleTag` and Concepts (`IsSystemLike`), grid-based collision detection exkurs

#### Runtime Systems
- **GameWorld**: Central resource container with type-indexed registries and `Session` management
- **Managers**: `HealthManager`, `ScorePoolManager`, `GameObjectPoolManager`, `SpawnManager`, `StateManager`, and others
- **State Management**: Generic `StateManager<TState>` template, `GameState` / `MatchState` hierarchical state machines, guards, hooks, state-to-viewport mapping
- **Messaging**: `TypedCommandBuffer` with fold-expression flush, `CommandHandlerRegistry`, double-buffered `EventBus` (Pass, Phase, Frame levels)
- **GameLoop**: Phase/Pass/System hierarchy with `CommitPoint` synchronization, `UpdateContext` as system parameter

### Benchmarks

The document includes a benchmark comparison of component operations (`HasComponent`, `GetComponent`, `GetMultipleComponents`, `HasComponent_Absent`) between the AoS implementation (Milestone 3) and the SoA implementation (Milestone 5):

| System | Layout | Throughput range (100–50,000 objects) |
|--------|--------|---------------------------------------|
| Desktop (Ryzen 9 9950X3D) | AoS (MS 3) | 18–81 M ops/s (declining) |
| Desktop (Ryzen 9 9950X3D) | SoA (MS 5) | 65–427 M ops/s (stable) |
| Laptop (i7-8750H) | AoS (MS 3) | 6–32 M ops/s (declining) |
| Laptop (i7-8750H) | SoA (MS 5) | 35–261 M ops/s (stable) |

The SoA implementation achieves 3–10× higher throughput with near-constant scaling across problem sizes.

### Performance Validation

FPS measurements with the *Runtime Stress Test Demo*:
- Desktop: >60 FPS sustained up to ~10,000 simultaneously rendered objects
- Laptop: >60 FPS sustained up to ~2,500 simultaneously rendered objects
- Identified bottleneck: scene graph iteration over inactive pooled objects

## Known Limitations

Unchanged from [v0.5.0-milestone5](v0.5.0-milestone5.md):

- Single-threaded execution model
- Developed and tested on Windows (MSVC 2022) only; Linux and macOS are untested
- Grid collision detection assumes uniform cell sizes
- Text rendering caches ASCII 0–127 only
- GameLoop uses variable time step (no fixed-timestep integration)
- Scene graph iterates over inactive pooled objects (performance bottleneck)
- No GPU-side optimizations (instancing, batching)
- No graphical effects for Game Feel enhancement (particles, grid deformation)
